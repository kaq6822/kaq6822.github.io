빌드 없는 정적 블로그의 발행 절차를 코드로 옮기기

2026-07-07

지금까지 이 블로그의 발행 절차는 전부 수동이었다. 템플릿을 복사하고, 토큰 다섯 개를 치환하고, 목록 페이지에 항목을 붙여 넣고, grep으로 잔존 토큰을 검사한 뒤 커밋한다. 개인 콘텐츠 파이프라인(RelayScribe)을 만들면서 이 마지막 단계를 자동화했다.

문제

RelayScribe는 Telegram으로 보낸 글감을 AI가 브리프로 정리하고 플랫폼별 초안까지 만들어 주는데, 정작 블로그 게시만 사람 손이 필요했다. 이 사이트는 Jekyll 같은 빌드 도구 없이 순수 HTML로만 구성한다는 원칙이 있어서, 흔한 "마크다운을 _posts에 밀어 넣는" 방식이 통하지 않았다.

선택지

  1. 사이트를 Jekyll로 전환 — 게시는 쉬워지지만 "빌드 도구 없음" 원칙을 포기해야 한다.
  2. 수동 패키지 유지 — 초안을 파일로 받아 사람이 붙여 넣는다. 자동화가 아니다.
  3. 발행 절차 자체를 코드로 — 사이트에 이미 문서화된 발행 계약(템플릿 토큰 치환, 목록 삽입, 잔존 토큰 검사)을 게시기가 그대로 수행한다.

결정 근거

발행 절차가 이미 저장소 안에 규칙으로 적혀 있었다는 점이 결정적이었다. 사람이 따르던 절차를 코드가 따르게 하면 사이트 쪽은 아무것도 바꿀 필요가 없다. 템플릿에 본문 삽입 지점 토큰 하나만 추가했고, 나머지는 게시기가 마크다운을 HTML로 바꿔 템플릿에 채우고 목록 페이지 최상단에 항목을 삽입한 뒤 커밋·push 한다.

결과

이 글이 그 파이프라인으로 발행된 첫 글이다. 손으로 하던 발행 절차(템플릿 복사부터 커밋까지 약 10분)가 버튼 한 번으로 끝난다. 정적 게시 모드에만 테스트 28개를 붙였고, 리뷰 과정에서 본문에 {{ site.url }} 같은 코드 예시가 있으면 잔존 토큰 검사가 오탐해 게시가 막히는 버그를 잡았다 — 기술 블로그에서 반드시 밟았을 케이스다. 아쉬운 점은 게시 전에 완성된 페이지를 미리 볼 수 없다는 것. 미리보기는 다음 과제로 남겼다.
